# ReAct Realm Refactorings: Analysis And Plan

Started: 2026-07-04 03:53 CEST

This entry maps the blast radius before implementation. The two required moves are coupled:

1. ReAct conversation-realm refs that are currently top-level (`fi:`, `ar:`, `tc:`, `so:`, `su:`, `ws:`) need canonical conversation-owned presentation, e.g. `conv:fi:...`, `conv:ar:...`.
2. Current-turn physical workspace names need to match their actual durability model:
   - old `turn_<id>/files/...` -> new `turn_<id>/git/projects/...`
   - old `turn_<id>/outputs/...` -> new `turn_<id>/files/...`
   - old `turn_<id>/snapshots/...` -> new `turn_<id>/git/snapshots/...`

The sensitive part is that these names are not just storage paths. They are model-facing protocol, persisted turn-log refs, named-service refs, artifact hosting refs, checkout roots, and git workspace pathspecs.

Single grammar policy: runtime implements only the new grammar. The new grammar
must have one meaning everywhere: harness, model
instructions, tool specs, UI, docs, and tests.

## Proposed Ref Grammar

**Design clarification (Elena, 2026-07-04) — do not conflate the two `conv` forms:**

- `conv:` is ONLY an owner-namespace **prefix** added in front of the currently
  orphaned ReAct namespaces (`fi:`, `ar:`, `tc:`, `so:`, `ws:`, `su:`) —
  producing `conv:fi:...`, `conv:ar:...`, etc.
- `conv_<conversation_id>` is a **path segment** with a different role —
  cross-conversation scoping inside the ref body — and **stays exactly where
  it is**. Both coexist in one ref:
  `conv:fi:conv_<conversation_id>.turn_<id>.files/<rel>`.

Any plan step that reads as "turn `conv_<id>` into the `conv:` namespace" is a
misreading; nothing about the segment changes.

This is the working target unless changed deliberately during implementation.

| Realm | Canonical Form | Meaning | Surfaces To Update |
| --- | --- | --- | --- |
| Conversation file/artifact | `conv:fi:turn_<id>.files/<rel>` or `conv:fi:conv_<conversation_id>.turn_<id>.files/<rel>` | Conversation-owned file artifact. After workspace rename, `files` means produced individual files. | Emitters, resolvers, instructions, docs, tests, chat/canvas/scene, descriptors. |
| Conversation authored row | `conv:ar:turn_<id>.user.prompt...` or `conv:ar:conv_<conversation_id>.turn_<id>.user.prompt...` | Conversation timeline/authored artifact row. | Emitters, resolvers, instructions, docs, tests. |
| Conversation tool call/result | `conv:tc:turn_<id>.<call>.call` / `.result` | Tool-call payload/result in conversation turn log. | Tool block builders, read/ref binding, instructions, docs, tests. |
| Conversation source pool | `conv:so:sources_pool[...]` or `conv:so:conv_<conversation_id>.sources_pool[...]` | Source pool rows attached to a conversation. | Source pool render/read/citation surfaces, instructions, docs, tests. |
| Conversation working summary | `conv:ws:turn_<id>.conv.working.summary` | Working summary artifact. | Working summary emit/read/search surfaces, instructions, docs, tests. |
| Conversation range summary | `conv:su:turn_<id>.conv.range.summary` | Compacted range summary artifact. | Compaction emit/read surfaces, instructions, docs, tests. |
| Skill refs | `sk:<skill_id>` until the skill subsystem owns a different canonical form. | Skill-resolution system, not conversation realm by default. | Existing `sk:`. |
| Owner refs | `mem:...`, `task:...`, `cnv:...`, etc. | External/named-service owner object refs. Pull or provider `object.get` decides materialization. | Existing owner refs. |

No pre-refactor form instructions, no pre-refactor form examples, and no alternate runtime branches.

## Proposed Physical Workspace Grammar

| Role | Old Physical Path | New Physical Path | New Logical File Ref |
| --- | --- | --- | --- |
| Durable project/workspace state | `turn_<id>/files/<scope>/...` | `turn_<id>/git/projects/<scope>/...` | `conv:fi:turn_<id>.git/projects/<scope>/...`. |
| Produced individual file/artifact | `turn_<id>/outputs/<scope>/...` | `turn_<id>/files/<scope>/...` | `conv:fi:turn_<id>.files/<scope>/...`. |
| Git-backed snapshots | `turn_<id>/snapshots/...` | `turn_<id>/git/snapshots/...` | `conv:fi:turn_<id>.git/snapshots/...`. |
| User attachments | `turn_<id>/attachments/...` | unchanged | `conv:fi:turn_<id>.user.attachments/...`. |
| External/pulled evidence | `turn_<id>/external/...` | unchanged unless later renamed | `conv:fi:turn_<id>.external...`. |

The logical ref path mirrors the physical role segment: `git/projects` and
`git/snapshots` are path segments in the ref body, not renamed to dotted tokens.
The parser/builder must treat role paths centrally so every surface uses the
same grammar.

## Risks

| Risk | Why It Matters | Mitigation |
| --- | --- | --- |
| Semantic drift between harness and model instructions | Any double meaning means ReAct may write/read the wrong path. | Update runtime, model instructions, tool specs, docs, and tests to the same new grammar. No conditional "old/new" wording. |
| Slash role segments in logical refs | Current helpers assume one namespace segment after `turn_<id>.`. | Refactor the parser/builder centrally to support `git/projects` and `git/snapshots` as first-class role path segments. |
| Model keeps producing obsolete paths | Instructions and tool specs currently teach old paths many times. | Remove obsolete examples completely once the runtime grammar is switched. |
| Conversation provider only handles files today | `conv:fi:` exists; other `conv:<ns>:` families are still emitted/resolved by ReAct internals. | Extend presentation/resolver helpers rather than invent separate named-service object kinds per family. |
| Sources/skills ownership ambiguity | `so:` is conversation source pool; `sk:` is skill subsystem. | Move `so:` under `conv:`; keep `sk:` with skill subsystem unless/until skill provider owns a named-service prefix. |
| Artifact hosting/RN paths | Hosted metadata may embed physical paths and old namespace names. | Inspect `conversation_store.py` and hosting code before changing emitted paths. |
