# AWS SM grouped secrets follow-up for deployment repo

This note is for the deployment agent working in `kdcube-internal-demo`.

## Scope

Align the deployment-side AWS Secrets Manager sync with the current KDCube runtime implementation.

This is required because KDCube runtime no longer treats AWS SM bundle secrets as one secret per leaf key.

## Runtime contract now

KDCube runtime now reads and writes AWS SM secrets in grouped documents:

- platform/global secrets:
  - `<prefix>/platform/secrets`
- bundle deployment secrets:
  - `<prefix>/bundles/<bundle_id>/secrets`
- user global secrets:
  - `<prefix>/users/<user_id>/secrets`
- user bundle secrets:
  - `<prefix>/users/<user_id>/bundles/<bundle_id>/secrets`

Examples for tenant `allcisodev` and project `example-product-ciso-dev`:

- `kdcube/allcisodev/example-product-ciso-dev/platform/secrets`
- `kdcube/allcisodev/example-product-ciso-dev/bundles/user-mgmt@1-0/secrets`

Inside the bundle document, the value is the bundle `secrets:` block itself.

Example:

```json
{
  "user_management": {
    "cognito_user_pool_id": "eu-west-1_xxx",
    "sheets_integration_credentials_file_content": "{...json...}"
  }
}
```

## Important compatibility note

Current KDCube runtime still has read fallback for:

- old leaf-per-secret AWS paths like:
  - `<prefix>/bundles/<bundle_id>/secrets/user_management/cognito_user_pool_id`
- current aggregate blob:
  - `<prefix>/bundles/secrets`

That fallback exists only for migration safety.

Deployment should now write the grouped bundle documents as the primary format.

## Required changes in `.github/scripts/sync_secrets.py`

### 1. Keep platform grouped doc write

Keep writing:

- `<prefix>/platform/secrets`

from:

- `SECRETS_YAML`

This already exists and is correct.

### 2. Change bundle secret writes

Stop treating bundle secrets as leaf secrets.

Current wrong behavior:

- writes many leaf secrets:
  - `<prefix>/bundles/<bundle_id>/secrets/<tail>`
- writes one aggregate blob:
  - `<prefix>/bundles/secrets`

Required new behavior:

- for each bundle in `BUNDLES_SECRETS_YAML`, write exactly one AWS secret:
  - `<prefix>/bundles/<bundle_id>/secrets`
- secret value must be the JSON serialization of that bundle's `secrets:` mapping

Example mapping from descriptor:

```yaml
bundles:
  version: "1"
  items:
    - id: "user-mgmt@1-0"
      secrets:
        user_management:
          cognito_user_pool_id: "eu-west-1_xxx"
          sheets_integration_credentials_file_content: "{...}"
```

must become:

- secret id:
  - `kdcube/allcisodev/example-product-ciso-dev/bundles/user-mgmt@1-0/secrets`
- value:

```json
{
  "user_management": {
    "cognito_user_pool_id": "eu-west-1_xxx",
    "sheets_integration_credentials_file_content": "{...}"
  }
}
```

### 3. Stop writing bundle leaf secrets

Remove leaf writing such as:

- `<prefix>/bundles/<bundle_id>/secrets/user_management/cognito_user_pool_id`

The current runtime no longer uses that as the primary storage model.

### 4. Aggregate blob

Target state:

- do not rely on `<prefix>/bundles/secrets`

For one migration cycle it is acceptable to keep writing it as a compatibility blob, but it must not be treated as the canonical deployment shape anymore.

Preferred final state:

- only grouped per-bundle documents

### 5. Do not touch user secrets

Deployment secret sync must continue to manage only deployment-scoped secrets.

Do not create, overwrite, export, or delete:

- user global secrets
- user bundle secrets

Those are runtime-owned.

## Workflow behavior

No change is required to the GitHub secret names for now:

- `SECRETS_YAML`
- `BUNDLES_SECRETS_YAML`

No change is required to descriptor staging for:

- `assembly.yaml`
- `gateway.yaml`
- `secrets.yaml`
- `bundles.yaml`
- `bundles.secrets.yaml`

The required change is in how `sync_secrets.py` writes AWS SM.

## Validation after patch

After secret sync, validate with AWS CLI.

```bash

  --secret-id kdcube/allcisodev/example-product-ciso-dev/platform/secrets \
  --region eu-west-1
```

Expected:

- one JSON document with platform/global secret structure

### Bundle secret doc

```bash

  --secret-id kdcube/allcisodev/example-product-ciso-dev/bundles/user-mgmt@1-0/secrets \
  --region eu-west-1
```

Expected:

- one JSON document
- includes:
  - `user_management.cognito_user_pool_id`
  - `user_management.sheets_integration_credentials_file_content`

## Current known customer descriptor issue

For ECS, bundle prop:

- `user_management.aws_profile`

should be empty.

ECS should use the task role, not a named local profile.

## Summary for the patch

Deployment repo must be changed so that:

- `SECRETS_YAML` -> `<prefix>/platform/secrets`
- each bundle in `BUNDLES_SECRETS_YAML` -> `<prefix>/bundles/<bundle_id>/secrets`
- leaf bundle secrets are no longer written as the primary format
- user secrets remain untouched
