# A
Short answer: yes, it’s reasonable, but it’s a medium refactor because today the services are wired around env vars. The least‑risk path is to keep env files for Docker/Compose
wiring, and let the app config itself come from assembly.yaml (with env as override/fallback).

Below is a concrete design + effort estimate, and then a design for bundle config/secrets.

———

## 1. “Assembly‑only” config for services (ingress/proc/metrics)

Goal
Mount assembly.yaml into containers and have services read config from it (instead of generating a large .env).

Recommended approach (safe + incremental)

1. Keep env files for Docker/Compose (ports, volumes, compose-only things).
2. Add a Settings loader that merges:
    1. Env overrides
    2. Assembly config (YAML)
    3. Defaults
3. Add a single config entry, e.g. KDCUBE_CONFIG_PATH=/config/assembly.yaml (already mounted).
4. Update get_settings() so it can read YAML and map to existing Settings fields.

Why
You can stop generating large envs later, but Docker still needs a small env set to start containers. This keeps changes localized and backwards compatible.

Effort
~2–4 days:

- Add YAML loader + mapping in Settings.
- Update services to rely on get_settings() for all config.
- Update docs + tests.

———

## 2. Bundle configuration & secrets (design)

You’re right: bundles need hierarchical config and secret config, and they can be updated at runtime via UI.

### Proposal

### A. Split bundle properties into config and secrets

In bundle metadata:

bundle:
config:
model: "gpt-4o-mini"
tools:
web_search: true
secrets:
openai:
api_key: null
stripe:
api_key: null

This mirrors assembly/secrets.

### B. Storage / resolution order

- Config stored in Redis/DB (current registry).
- Secrets stored in secrets store (sidecar or external).
- Runtime resolution order:
    1. Operational overrides (bundle admin UI)
    2. Assembly defaults
    3. Bundle defaults

### C. Bundle admin UX

- Non‑secret fields: normal editable inputs.
- Secret fields: show “configured / not set”, never show value.
- On submit, secrets go to a secret set endpoint, config goes to the bundle config endpoint.

### D. API split

- PUT /integrations/bundles/{id}/config  → Redis/DB
- PUT /integrations/bundles/{id}/secrets → secrets sidecar (via proc)

### E. Secrets sidecar integration

- UI never talks to sidecar directly.
- Proc accepts secret updates, uses its admin token to set secrets in sidecar.
- Secret keys use dot‑path names:
  bundles.<bundle_id>.secrets.openai.api_key

This matches your dot‑path strategy and keeps secrets out of Redis/DB.

Local deployment: sidecar in‑memory only (secrets are re‑entered after restart).
Production: replace sidecar backing with external secret manager later.

———

## 3. Deployment vs operational overrides

Make it explicit:

Config precedence

1. Runtime override (bundle admin or gateway updates)
2. Assembly descriptor
3. Bundle defaults

Secrets precedence

1. Runtime secret (bundle admin / secrets sidecar)
2. Secrets descriptor (if allowed in that environment)
3. None

———

## 4. What changes to do now (minimal to unblock)

If you want the lowest‑effort step now:

1. Add KDCUBE_CONFIG_PATH to services (ingress/proc/metrics).
2. Extend get_settings() to read YAML (assembly) and map into Settings.
3. Keep env for compose + proxylogin + infra.
4. Add bundle secrets API and model later.
